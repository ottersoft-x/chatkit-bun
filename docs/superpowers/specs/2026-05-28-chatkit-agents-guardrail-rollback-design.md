# ChatKit Agents Guardrail Rollback Design

## Purpose

This milestone adds rollback semantics to `streamAgentResponse(...)` when the
JavaScript Agents SDK raises a guardrail tripwire while a ChatKit response is
streaming. ChatKit should remove items already produced by the current agent
turn so the client and store do not retain unsafe, partial, or abandoned output.

The behavior mirrors the Python `chatkit.agents.stream_agent_response(...)`
guardrail cleanup while accounting for the JavaScript SDK's additional tool
guardrail tripwire classes.

## Scope

This milestone includes:

- Tracking item ids produced by `streamAgentResponse(...)` during the current
  turn.
- Emitting `thread.item.removed` events for produced item ids when a guardrail
  tripwire is raised.
- Rethrowing the original guardrail tripwire error after cleanup events are
  yielded.
- Closing the context event queue and returning both SDK and context iterators
  during guardrail cleanup.
- Focused tests for SDK-produced items, context-produced items, and JavaScript
  input, output, tool input, and tool output guardrail tripwire classes.

This milestone does not include:

- Rollback for non-guardrail stream errors.
- New public options for deciding which errors trigger rollback.
- Changes to `ChatKitServer.processEvents(...)` persistence rules.
- New thread stream event schemas.
- Changes to `ThreadItemConverter`, input replay, widget serialization, or
  generated-image URL conversion.

## Current Behavior

`streamAgentResponse(...)` currently converts SDK and context events into
`ThreadStreamEvent` values. It yields item additions, updates, and done events as
they arrive, then closes iterators in a `finally` block.

When the SDK stream throws, the error currently escapes through the generator
without first emitting removal events for items already produced by the current
turn. At the server layer, generic responder errors are converted to a retryable
`error` event, but the agent bridge itself does not distinguish guardrail
tripwires from other failures.

The Python reference keeps a `produced_items` set in `stream_agent_response(...)`.
It adds ids for ChatKit items emitted by context helpers, assistant messages,
reasoning workflows, generated images, and SDK tool call metadata. On input or
output guardrail tripwire, it yields `thread.item.removed` for each produced id,
drains remaining context events without processing them, and rethrows the
original guardrail exception.

## Desired Behavior

`streamAgentResponse(...)` should maintain a turn-local set of produced item ids.
An id is produced when this bridge yields an item-bearing event or learns about a
tool-call item id that can later be surfaced as a ChatKit client tool call.

The bridge should track:

- `thread.item.added` and `thread.item.done` events emitted from
  `AgentContext.stream(...)` and helper methods such as `streamWidget(...)`.
- Auto-ended workflow `thread.item.done` events inserted by workflow lifecycle
  handling.
- Assistant message ids created from SDK `response.output_item.added`,
  `output_text_delta`, normalized `output_text_delta`, `response_done`, and
  `response.output_item.done` flows.
- Reasoning workflow ids created from SDK reasoning events.
- Generated image item ids created from SDK image generation events.
- SDK tool call item ids discovered from `run_item_stream_event` tool-call
  metadata when available.
- Fallback client tool call ids generated by `pendingClientToolCallEvent(...)`
  if a client tool call is yielded before a guardrail error.

If a guardrail tripwire is thrown by the SDK stream, by context event handling,
or while converting a guardrail-producing SDK event, the bridge should:

1. Yield a `thread.item.removed` event for each produced item id in insertion
   order.
2. Close the context event queue so queued app events are not processed after the
   tripwire.
3. Return the SDK iterator and context iterator through their `return(...)`
   methods when available.
4. Rethrow the original guardrail tripwire error.

The rollback trigger should include all JavaScript Agents SDK guardrail
tripwire classes exported from `@openai/agents`:

- `InputGuardrailTripwireTriggered`
- `OutputGuardrailTripwireTriggered`
- `ToolInputGuardrailTripwireTriggered`
- `ToolOutputGuardrailTripwireTriggered`

Non-guardrail errors should keep the current behavior. The bridge should not
emit removal events for ordinary conversion failures, iterator failures,
application errors, or cancellation paths unless those errors are instances of
the guardrail tripwire classes above.

## Architecture

Add a small guardrail helper inside `src/agents/stream.ts`:

- `isGuardrailTripwire(error: unknown): boolean`
  - Uses `instanceof` checks against the four guardrail tripwire classes imported
    from `@openai/agents`.
  - Stays private to the stream module unless another module later needs the
    behavior.

Add a produced-item tracker inside `streamAgentResponse(...)`:

- Use `Set<string>` to preserve insertion order and dedupe repeated adds/done
  for the same item.
- Track ids immediately before yielding parsed item-bearing events.
- Track SDK tool metadata ids when `toolCallMetadata(...)` returns an `itemId`.
- Track pending client tool call ids before yielding the final pending tool call
  event.

Add a rollback path around the existing streaming loop:

- Keep the existing `finally` cleanup for normal completion, cancellation, and
  non-guardrail errors.
- Add a `catch` for errors from the streaming loop and finalization block.
- If the caught error is a guardrail tripwire, yield removal events for tracked
  ids and rethrow.
- If the caught error is not a guardrail tripwire, rethrow without removal
  events.

The implementation should avoid double cleanup. The context queue should still
be closed exactly once, and iterator `return(...)` should remain best-effort.

## Data Flow

Normal stream:

1. Load recent thread items and resume workflow state.
2. Merge SDK and context iterators.
3. Convert events and yield ChatKit thread stream events.
4. Track produced ids for item-bearing events.
5. Persist open workflow at normal completion.
6. Yield pending client tool call if present.
7. Close iterators.

Guardrail stream:

1. Load recent thread items and resume workflow state.
2. Merge SDK and context iterators.
3. Convert and yield some ChatKit events while tracking produced ids.
4. SDK, context handling, or conversion throws a guardrail tripwire.
5. Yield `thread.item.removed` for tracked ids.
6. Close context events and return iterators.
7. Rethrow the original guardrail tripwire.

## Error Handling

Guardrail errors are not converted into `stream.error` inside
`streamAgentResponse(...)`. The original error must rethrow so the caller can
emit custom safety messaging, as shown in the Python guide.

Removal event parsing should use `ThreadStreamEventSchema.parse(...)`, matching
the rest of the bridge. If parsing ever fails, that failure should surface rather
than silently suppressing cleanup corruption.

The order of removal events should be deterministic. The first produced item id
is removed first, matching `Set` insertion order. Tests should compare sets when
ordering is not behaviorally important and compare arrays when deterministic
cleanup order is part of the assertion.

## Testing

Add focused tests in `tests/agents.test.ts`:

- A parameterized test for the four JavaScript guardrail tripwire classes.
- A test where the SDK produces an assistant message, a context helper produces
  an assistant or widget item, and a guardrail tripwire occurs. The expected
  output includes removal events for every produced id and the original error is
  rethrown.
- A test where SDK tool call metadata is observed before a guardrail tripwire.
  The metadata item id is removed even if no ChatKit client tool call event has
  been yielded yet.
- A test where a non-guardrail error occurs after produced items. No removal
  events are yielded, preserving existing behavior.
- A test that queued context events after a guardrail are not yielded.

Existing server tests for `thread.item.removed` persistence should remain
unchanged. Full verification should use:

```bash
bun test tests/agents.test.ts
bun run typecheck
bun run verify
```

## Acceptance Criteria

- `streamAgentResponse(...)` removes all items produced by the current turn when
  any JS Agents guardrail tripwire class is thrown.
- The original guardrail tripwire error is rethrown after cleanup events.
- Non-guardrail stream failures do not emit rollback removal events.
- Context events queued after a guardrail are not processed.
- `ChatKitServer.processEvents(...)` continues to persist `thread.item.removed`
  events through its existing deletion path.
- No public schema, server request, converter, or widget API changes are added.
- `bun run verify` passes.
